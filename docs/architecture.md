# CLI-Campus 架构白皮书

> "让校园生活重回命令行，构建 AI 时代的数字校园基础设施。"
> —— I++ Open Source Culture Club 孵化项目

## 1. 项目愿景与核心定位

CLI-Campus 旨在彻底解决东南大学（及未来更多高校）碎片化、异构化的校园 Web 系统带来的糟糕体验。

它**不是**一个简单的爬虫脚本集合，也**不是**一个大包大揽的通用 AI 助手。它的终极定位是：

> **校园数字基建的"系统调用" (System Call) 与 Agent-Native 的底层武器库。**

- **向下**：通过极其鲁棒的适配器（Adapter）网络，抹平教务处、财务处、学习通等平台的历史包袱和反爬限制。
- **向上**：提供干净、极速、结构化（JSON）的标准 API，成为任何 AI Agent、微信 Bot 或快捷指令的首选底层依赖。

## 2. 架构演进路线

### 2.1 初始构想：烟囱式脚本 (V0)

- **形态**：为每个功能（查课表、查校车）编写独立的爬虫和 CLI 解析逻辑。
- **痛点**：高耦合。一旦教务系统前端改版，整个 `seu course` 命令直接崩溃；数据源混杂（PDF、网页、API），维护成本呈指数级上升。

### 2.2 最小实践架构：Adapter-Provider 模式 (MVP)

- **The Core (核心层)**：极其轻量，基于 `Typer` 构建。只负责接收参数、调用 Adapter、以及通过 `Rich` 渲染 UI 或输出 JSON。
- **The Provider (适配器层)**：负责一切"脏活累活"。将不同平台的数据洗净后，转化为全局统一的 Pydantic 模型。
- **The Vault (状态层)**：利用操作系统原生 Keyring 管理统一身份认证（CAS）凭证。

### 2.3 终极形态：Agent-Native 多租户基座 (V2.0+)

- **原子化工具库**：每个 Adapter 方法都是最小粒度的操作。
- **宏指令（SOP）**：预设 YAML 编排的标准操作流程。
- **多校园命名空间**：针对软件供应商而非单所学校开发 Adapter，通过 `campus init` 初始化配置文件，瞬间拓展至全国各大高校。

### 架构对比一览

| 维度 | 初始架构 (V0) | 终极架构 (Agent-Native V2) |
| :--- | :--- | :--- |
| **数据流向** | 爬虫直连终端 | Adapter → 统一 Schema → JSON/UI |
| **功能粒度** | 宏大且固化 | 原子化工具组合 |
| **AI 参与度** | 无 | SOP 固定编排 + LLM 动态编排 |
| **生态兼容** | 仅限东大特定系统 | 基于供应商开发，多高校通用 |
| **调用方** | 仅限终端人类用户 | 人类 + LLM Agent |

## 4. 护城河与生态位

### 不重合的生态位

- **通用 Agent（如 CLI-Anything）** 是"大脑"，负责将自然语言翻译为指令。
- **CLI-Campus** 是"手脚"，是高度专业化的领域武器库。
- 当用户输入"帮我查东大课表"时，通用 Agent 无法搞定 CAS 动态加密，但它可以无缝调用 `campus course --json`。

### 绝对的领域壁垒

全世界只有本土开源团队，才知道校车时刻表隐藏在哪个 PDF 节点，知道不同校区（九龙湖/四牌楼/丁家桥）的业务潜规则。

### 供应商红利

国内高校高度依赖同质化的 SaaS 供应商（正方、强智、超星）。攻克一个供应商接口，即可辐射数十所高校。
